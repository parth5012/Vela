# Long-Term Semantic Memory Design

## 1. Context
To enable the supervisor agent to store and recall personal details, projects, preferences, and demographics, we need to implement the long-term semantic memory module. This leverages PostgreSQL's `pgvector` extension to store 768-dimensional embeddings generated by Gemini's `text-embedding-004` model.

## 2. Proposed Changes

### 2.1 Context Retrieval (`agent/graph.py`)
- Implement `build_context(state)`:
  - If `db_conv_id` is present and valid, generate a 768-dimensional embedding for the latest user message.
  - Query the `memory_vectors` table for the top 3 closest memories (using cosine distance) for the active `conversation_id`.
  - Format these memories and return them to be injected into the system prompt.

### 2.2 Memory Extraction, Deduplication & Storage (`agent/graph.py`)
- In `chatbot_node`, after the response is generated, trigger a background thread to process memory extraction:
  1. **Extraction**: Ask the LLM to identify permanent facts, preferences, or project details from the current conversation turn.
  2. **Similarity Lookup**: For each extracted fact, generate its embedding and query the database for any highly similar existing memory (threshold cosine distance < 0.35).
  3. **Reconciliation**: If a similar memory exists, ask the LLM to classify the relationship:
     - `DUPLICATE`: Do nothing (skip insertion).
     - `CONFLICT`/`SUPERSEDE`: Delete the old memory and insert the new one.
     - `REFINEMENT`: Update the text and embedding of the old memory.
     - `NONE`: Insert the new memory.
  4. **Save**: Save new/updated memories to the `memory_vectors` table.

## 3. Detailed Design

### 3.1 LLM Prompts

#### 3.1.1 Memory Extractor Prompt
```
Analyze the following user-agent interaction. Extract any permanent, factual information about the user, including their name, demographics, educational status, career goals, project details, coding stack, preferences, or relationships.

Input:
User: {user_query}
Agent: {agent_response}

Instructions:
- Output a JSON list of strings, where each string is a single, clear, independent fact.
- Example: ["Completed Class 12 in 2024", "Wants a backend internship for Summer 2026"]
- Do not extract temporary conversational content (e.g. questions, jokes, greetings).
- If no permanent facts are found, return an empty list.
- Return ONLY the JSON list. No explanation or formatting outside the JSON.
```

#### 3.1.2 Reconciliation Prompt
```
You are a Memory Reconciliation System. Compare a newly extracted fact with an existing memory and determine if the new fact contradicts, duplicates, refines, or is completely independent of the existing memory.

Existing Memory: "{existing_memory}"
New Fact: "{new_fact}"

Classification Rules:
- DUPLICATE: The new fact says the exact same thing as the existing memory.
- CONFLICT: The new fact contradicts the existing memory (e.g., year, name, or goal has changed).
- REFINEMENT: The new fact provides more detail or updates the existing memory without direct contradiction.
- NONE: The new fact is unrelated/independent.

Return ONLY one of the following classification labels: DUPLICATE, CONFLICT, REFINEMENT, NONE.
```
